# 건축법규 검토 RAG 시스템 — 4-Layer 통합 파이프라인 계획 (v2.1)

## 1. 프로젝트 개요

### 1.1 목표

기존 **국가법령 + 질의회신 RAG** 시스템에 **입법취지(Revision Reason)** 와 **판례(Court Case)** 레이어를 추가하여, 4개 레이어에서 검색된 근거를 하나의 논리 구조로 연결하는 고도화된 법률 어시스턴트를 구축한다.

### 1.2 최종 아키텍처 (4-Layer Hybrid Reasoning)

```
[사용자 질문]
    │
    ▼
╔══════════════════════════════════════════════════════════════════╗
║  1차 LLM 호출 — 쟁점 식별 + 검색 트리거 추출                        ║
║                                                                  ║
║  출력 ①: 핵심 쟁점 (1~2문장)                                       ║
║  출력 ②: 검색 트리거 키워드 (사유 태그 포함)                          ║
║  출력 ③: 예상 관련 법령 힌트 (법명, 조문번호)                         ║
║                                                                  ║
║  ※ 관계 유형(7-Types)은 Pass 1에서 분류하지 않는다.                 ║
║    유형은 판례 수집 단계(13_CaseLabeler.py)에서 정적 라벨로 부여한다.  ║
╚══════════════════════════════════════════════════════════════════╝
    │
    ▼ [3-Layer 라우팅: 토픽→키워드맵→조문그래프]
    │
    ├──────────────────────────────────────────────────────┐
    │                                                      │
    ▼ [전략 2: 벡터 공간 주입]             ▼ [전략 1: 그래프 확장]
┌──────────────────────┐           ┌──────────────────────────────┐
│ Law_Articles         │           │ Explicit Reference Graph     │
│ (조문 + 참조정보 임베딩) │ <──────> │ (건축법 ↔ 타법령 명시적 참조)    │
└──────────────────────┘           └──────────────────────────────┘
    │           │                      │                │
    ▼           ▼                      ▼                ▼
[1층: 조문]  [2층: 입법취지]         [3층: 질의회신]    [4층: 판례]
 법규 필터     원자적 청킹 검색          유사도 검색      (법규×유형) 필터
 현행         ↔ 조문 ID 연결           CoT 선례        7-Type 정적 라벨
 ✅ Active    ⏳ Phase 1              ✅ Active        ⏳ Phase 3
    │           │                      │                │
    └───────────┴──────────────────────┴────────────────┘
                                │
                                ▼
[2차 LLM 호출] → 4개 레이어 통합 기반 법리 답변
  컨텍스트 = [조문] + [입법취지] + [질의회신 CoT 선례] + [판례]
```

**핵심 설계 원칙**:
- **Pass 1 단순화**: 유형 분류를 Pass 1에서 제거. 오류 전파를 차단하고 응답 속도 향상.
- **정적 라벨 우선**: 관계 유형은 판례 라벨링 단계에서 1회 부여 → 검색 시 메타데이터 필터로 활용.
- **CoT 선례 활용**: 질의회신 선례가 논리 구조 자체를 담고 있으므로 Pass 2가 이를 직접 참조.

## 2. 입법취지 레이어 구축 (Phase 1 — New)

### 2.1 데이터 소스 및 전처리

- **소스**: 국회 검토보고서 및 국가법령정보센터 '제·개정이유' (3만 페이지 분량)
- **원자적 청킹(Atomic Chunking)**: '◇ 주요내용' 하위의 항목(가, 나, 다...) 단위로 분할
- **동적 맥락 결합**: 각 항목별 텍스트에 최상단 '◇ 개정이유(배경)'를 결합하여 정보 완결성 유지

### 2.2 메타데이터 태깅

| 필드 | 추출 방법 |
|------|---------|
| `article_id` | 텍스트 내 괄호 `(제00조)` 패턴에서 추출 |
| `law_name` | 법령명 헤더에서 추출 |
| `enforcement_date` | 시행일 및 대통령령/법률 번호 |
| `purpose_keyword` | 규제 완화 / 절차 간소화 / 안전 강화 등 입법 목적 키워드 |

### 2.3 벡터 공간 주입 전략 (Indexing)

조문 임베딩 시 해당 조문이 참조하는 타 법령(External Reference) 정보를 텍스트 상단에 주입 → 벡터 공간 내 관련 법령들이 군집화되도록 유도.

## 3. Generator 통합 — Pass 구조

### 3.1 Pass 1 (현재 구조)

```
입력: 사용자 질문
출력:
  ① 핵심 쟁점 (1~2문장)
  ② 검색 트리거 키워드:  ※ "키워드" [사유: 정의참조|요건위임|...]
  ③ 법령 힌트 JSON:      {"law_hints": ["건축법 제19조", ...]}
```

> **Pass 1에서 관계 유형(7-Types)을 제거한 이유**:
> - 분류 오류가 검색 top_k에 직접 전파됨
> - 유형 경계가 모호해 분류 자체가 noise가 될 수 있음
> - CoT 선례가 논리 구조를 암묵적으로 포함하므로 명시적 라벨 불필요
> - 판례 라벨링 시 정적으로 부여하면 충분

### 3.2 Pass 2 컨텍스트 포맷

```
=== [관련 법령 조문] ===
[1] 건축법 제2조 (정의)
{조문 내용}

=== [입법취지] ===                      ← Phase 1 완료 후 추가
[취지1] 건축법 제2조 개정 (2023-00-00)
{원자적 청킹 내용 + 개정 배경}

=== [유사 질의회신 선례] ===
아래는 유사 질의에 대한 CoT 선례입니다.
사실관계가 본 건과 다른 경우 직접 인용하지 말고 논리 구조만 참고하세요.

[선례1] [서울시건지 88550-2257 / '13.03.07] (2013-03-07) [seoul 2015]
질문: {선례 질문}
{CoT 답변 내용}

=== [참조 판례 풀] ===                  ← Phase 3 완료 후 추가
아래는 (법규 × 유형) 쌍으로 검색된 관련 판례입니다.
법리 해석 중 조문만으로 판단이 애매한 지점에서만 인용하세요.
확장된 정의를 원용할 경우, 해당 판례의 조건이 본 건과 일치하는지 반드시 검토하세요.

[판례1] 대법원 2023두12345 (2023.5.15. 선고)
  유형: 정의확장형 | 법규: 건축법 제2조제1항제1호
  요지: "대지"의 범위에 분할예정 토지 포함
  적용 조건: 사용승인 시 분할이 예정된 경우에 한함
```

### 3.3 Pass 2 답변 구조

```
[쟁점 식별] → [결론 (확신도)] → [관련 조문 확인]
→ [이유: 문언해석 / 체계적 해석 / 목적·취지 해석]
→ [유사 질의회신 검토]    ← CoT 선례 논리 참조
→ [관련 판례 검토]        ← Phase 3 이후
→ [근거 법령 + 인용 선례] → [담당부서 확인 질문 (확신도 ≠ 확정)]
```

## 4. 판례 레이어 구축 (Phase 3)

### 4.1 관계 유형 7-Types (판례 정적 라벨)

| 유형 | 해석 분기점 | 판례 활용 방식 | 입법취지 결합도 |
|------|-----------|---------------|--------------|
| `DEF_EXP` | 용어 정의 확장 여부 | 확장된 정의 범위 제시 | 높음 (개정 의도 확인) |
| `SCOPE_CL` | 적용 범위 경계 확인 | 확정된 적용 경계 제시 | 중간 |
| `REQ_INT` | 요건 충족 기준 모호 | 구체화된 판단 지표 제시 | 높음 (기준 신설 배경) |
| `EXCEPT` | 예외 인정 가능성 | 인정된 예외 조건 제시 | 최상 (특례 신설 이유) |
| `INTER_ART` | 조문 간 충돌/우선순위 | 확립된 우선순위 제시 | 중간 (타법 관계 확인) |
| `PROC_DISC` | 행정청 재량 범위 | 확인/제한된 범위 제시 | 중간 |
| `SANC_SC` | 제재/벌칙 비례성 | 구체화된 부과 기준 제시 | 중간 |

> 유형은 13_CaseLabeler.py에서 판례 1건당 1회 부여. 검색 시 `relation_types` 메타데이터 필터로 사용.

### 4.2 판례 검색 전략 — (법규 × 유형) 쌍 필터링

```
법령 힌트 (Pass 1 출력):
  law_hints = ["건축법 제2조", "건축법 제11조"]

판례 검색 (05_Retriever.py — search_cases_by_type):
  ① 1차: cited_laws_str ∋ "건축법" AND relation_types ∋ "DEF_EXP"
  ② Fallback 1: 유형 필터 완화 → 법규 필터만
  ③ Fallback 2: 법규 필터도 완화 → 전체 검색

복수 쌍 병합:
  - 각 유형별로 독립 검색 후 RRF 재병합
  - 중복 판례는 최고 가중 점수 채택
  - weight ≥ 0.5인 유형만 필터링에 사용
```

### 4.3 판례 활용 3가지 모드

| 모드 | 이름 | 매칭 조건 | 답변 내 위치 |
|------|------|----------|------------|
| ① | **직접 적용** | 법령 ✅ 유형 ✅ 키워드 ✅ | `[관련 판례 검토]` 별도 섹션 |
| ② | **논리 차용** | 법령 ✅ 유형 ✅ 키워드 ❌ | `[법리적 판단]` 내 인라인 |
| ③ | **정의 원용** | 선행 판례의 정의가 전제 조건 | `[관련 조문 확인]` 직후 |

## 5. 확신도 판정 및 담당부서 확인 질문

### 5.1 확신도 4단계

| 확신도 | 의미 | 시스템 행동 |
|--------|------|-----------|
| **확정** | 법+선례로 결론 명확 | 확정 답변 제공 |
| **조건부** | 사실관계에 따라 결론 상이 | 조건별 답변 + 확인 질문 |
| **재량위임** | 허가권자 재량 사항 | 법적 근거 제시 + 확인 질문 |
| **범위외** | 시스템 검토 범위 밖 | 관할부서 안내 + 확인 질문 |

### 5.2 "담당부서 확인 질문" 트리거 5가지

1. **재량 판단 영역** — 법/선례로 답이 정해지지 않는 허가권자 재량
2. **현장별 사실관계 필요** — 시스템이 알 수 없는 현장 정보
3. **관련 조문·선례 부재** — 검색 결과 없음
4. **선례 충돌** — 상반된 선례 존재
5. **복합 조건 미해결** — 분기 조건이 너무 많아 확정 불가

> **핵심**: 단순히 "문의바랍니다"가 아니라, **어떤 질문을 왜 해야 하는지**까지 제시. 각 질문 아래 `→`로 그 질문이 답변에 미치는 영향을 설명.

## 7. 구현 단계 및 우선순위

### Phase 1 — 입법취지 레이어 (데이터 정제)

```
├─ '전체 제개정이유.txt' 파싱 스크립트
│   ├─ '◇ 주요내용' 하위 가/나/다 항목 원자적 분리
│   └─ 조문 번호 기반 자동 메타데이터 태깅
└─ 입법취지 컬렉션 → Chroma revision_reasons 인덱싱
```

### Phase 2 — 검색 엔진 통합

```
├─ Rich Chunk 기반 벡터 검색 구현
├─ 명시적 참조 기반 그래프 확장 로직 (1-Hop) 연동
└─ 입법취지 레이어 메타데이터 필터링 검색 추가
```

### Phase 3 — 판례 파이프라인

```
├─ 11~15 스크립트로 판례 수집·파싱·라벨링·인덱싱
├─ 05_Retriever.py: court_cases 검색 활성화 (현재 코드 준비 완료)
└─ 06_Generator.py: Pass 2 컨텍스트에 판례 섹션 추가
```

### Phase 4 — 검증 및 최적화

```
├─ 엔드투엔드 테스트: 10개 질문 → 4-Layer 답변 품질 평가
├─ 검색 파라미터 튜닝 (top_k, RRF 가중치)
└─ 확신도 판정 정확도 평가
```
