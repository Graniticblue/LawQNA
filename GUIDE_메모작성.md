# 메모 작성 가이드

메모(memo)는 해석례·판례 원문에서 직접 읽히지 않는 **분석적 부가가치**를 축적하기 위한 공간이다.  
원문 요약이 아니라, 원문을 읽은 후에야 보이는 것—법리 구조, 반복 패턴, 논거 함정—을 기록한다.

---

## 1. 메모를 써야 할 때 vs 쓰지 않아도 될 때

### 써야 할 때

| 상황 | 예시 |
|------|------|
| 여러 해석례에서 반복되는 패턴을 발견했을 때 | 동일 문언 전용 시도 (memo_004) |
| 논거 구조에 함정이나 전제 붕괴가 있을 때 | 엄격해석 전제 붕괴 (memo_005) |
| 두 해석례가 합산돼야 완성되는 법리일 때 | 가목·나목 OR + 건축물 없을 것 (memo_009) |
| 원문에 "향후 보완 필요"한 미완 분석이 있을 때 | 그 밖의 원문 입수 전 (memo_006 구버전) |
| AI가 반복적으로 오독하는 조문 효과가 있을 때 | 제6항 자동 이동 효과 간과 (memo_005 끝) |
| 입법기술 일반론으로 다른 조문에도 적용 가능할 때 | '그 밖의' 표현 (memo_006) |

### 쓰지 않아도 될 때

- 해석례 원문을 그대로 요약만 하는 경우 → 청크(precedents_2026_april)로 충분
- 결론이 조문 문언에서 바로 읽히는 단순 적용 사례
- 이미 동일 내용을 다루는 메모가 있는 경우 → 기존 메모에 보완

## 2. 메모 유형

### A. 단일 해석례 분석
특정 해석례에 대해 원문이 설명하지 않은 규범적 근거나 보완 이론을 추가한다.

```
예: memo_002 — 22-0361에서 법제처가 설명하지 않은 허가 심사의 법적 성질
    memo_003 — 21-0674에서 10분의 1이 '변화폭 측정 기준'임을 분석
```

### B. 반복 패턴
여러 해석례에 걸쳐 나타나는 공통 논리 구조를 추상화한다. `linked_to`에 복수 코드.

```
예: memo_004 — 동일 문언 전용 시도 패턴 (21-0640, 21-0674, 04-0663)
```

### C. 논거 구조 / 함정 경고
"이렇게 추론하면 틀린다"는 패턴. 전제 붕괴, 조문 오독 경로 포함.

```
예: memo_005 — 엄격해석 원칙이 틀린 결론을 낳는 조건
```

### D. 연대 해석례
두 해석례가 독립적으로는 부분 규율이지만 합산하면 법리가 완성되는 경우.

```
예: memo_009 — 20-0620(건축물 없을 것) + 21-0054(OR 관계) = 제6항제2호 완성
```

### E. 입법기술 일반론
특정 사안을 넘어 다른 조문 해석에도 적용 가능한 범용 원칙.

```
예: memo_006 — '그 밖의/그 밖에' 표현의 보충적 포괄 기능
```

## 3. 파일 구조

### memos.jsonl (기계 처리용)

```json
{
  "memo_id": "memo_013",
  "linked_to": ["해석례코드1", "해석례코드2"],
  "title": "[코드] — [핵심 쟁점 한 줄]",
  "content": "평문 전문 (줄바꿈은 \\n으로)",
  "tags": ["태그1", "태그2"],
  "created": "YYYY-MM-DD"
}
```

### memos_md/{memo_id}.md (사람이 읽는 버전)

```markdown
# {memo_id} — {title}

**linked_to:** 코드1, 코드2
**tags:** 태그1, 태그2, ...
**created:** YYYY-MM-DD

---

본문
```

## 4. 필드 작성 요령

### memo_id
`memo_NNN` 형식, 3자리 제로패딩. 현재 최신: `memo_012`.

### title
- 형식: `[해석례코드] — [핵심 쟁점]`
- 반복 패턴 메모는 코드 없이 패턴명으로: `반복 패턴 — 동일 문언 전용 시도, ...`
- 입법기술 일반론: `'표현' 입법기술 원칙 — 출처 해석례`

### linked_to
- 관련 해석례 코드 배열. 없으면 `[]`
- 법제처 형식: `"09-0041"`, `"22-0361"` 등

### tags
검색에 걸려야 할 키워드. 아래 4가지를 조합:
1. **조문번호**: `제86조제2항제1호`, `건축법제52조`
2. **쟁점 유형**: `엄격해석전제검증`, `동일문언전용`, `명시적제외규정`
3. **해석례 코드**: `22-0361`, `15-0082`
4. **법리 키워드**: `보충적포괄`, `개방형열거`, `OR관계`

### content (jsonl용)
- `\n`으로 단락 구분
- `【제목】` 형식으로 소제목 표시
- MD 문법(`**`, `##`) 사용하지 않음 — 평문으로

## 5. 본문 작성 원칙

### 해야 할 것
- **결론을 첫 단락에** — 읽는 이유가 분명해야 함
- **조문 번호 명시** — `제86조제2항제1호` 수준으로 구체적으로
- **비교가 필요하면 표** — 두 가지를 대조할 때 산문보다 표가 명확
- **활용/처방 명시** — "이런 상황에서 이렇게 쓴다"로 마무리
- **AI 오독 경로 포함** (해당하는 경우) — 어떤 프레임이 오류를 유발하는지

### 하지 말아야 할 것
- 원문 이유 섹션을 그대로 옮기는 것 → 청크로 충분
- "~라고 규정하고 있다" 식의 조문 나열
- 결론 없이 쟁점만 기술
- 하나의 메모에 무관한 여러 주제 혼재 → 메모를 분리

## 6. 인덱싱

메모를 추가·수정한 후 반드시 재인덱싱:

```bash
python ingest_memos.py
```

전량 재인덱싱 방식이므로 jsonl만 맞으면 된다.

---

## 7. 중복 검토

새 메모를 쓰기 전에 기존 메모와 겹치는지 확인:

1. `data/memos_md/` 파일명 훑기
2. 같은 해석례 코드(`linked_to`)가 이미 있는지 확인
3. 겹치면 **새 메모 생성 대신 기존 메모에 섹션 추가**

> memo_006이 "원문 확인 후 보완 필요" 상태로 있다가 15-0082 원문 입수 후 보완된 것이 좋은 사례.
