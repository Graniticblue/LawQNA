## 0. 절대 규칙 (위반 시 사용자가 반드시 잡아냄 — 실제 전례 있음)

1. **공동 판정**: ⑥b 측정 이후의 판독·반영·포지션 판정은 **사용자와 공동**으로 한다.
   AskUserQuestion으로 제안(추천 표기 가능)하고 답을 받은 뒤 확정한다. 단독 확정 금지.
2. **판단 필드 수동**: gist·relation_name·position·doctrine_terms·계열명은 매 사안 사람이 읽고 판단.
   헬퍼(curate_lib)는 기계 검증만. LLM 일괄 생성 금지.
3. **케이스패치 금지**: 메커니즘 보수(⑥b′)는 전역 이득 수정만 — 특정 질의어 하드코딩,
   특정 doc 부스팅, eval 통과만을 위한 조작은 전부 금지. "이 수정이 다른 질의에도 이득인가?"로 검문.
4. **대장 밖 학습 금지**: 모든 학습(라이트트랙 포함)은 `학습후보_체크리스트.md`에
   등재 → 완료 표기 → eval 후보 등재의 3단 장부를 거친다.
5. **절차 생략 금지**: 배치 처리·부채 청산이라도 eval 5단계(§3)는 건별 전부 밟는다.
   "간소화된 것 같은데 누락 없냐"는 지적이 실제로 나왔고, 실제로 누락이 있었다(20-0008).

## 3. eval 트랙 표준 5단계 (측정 요청·부채 청산 시 — 건별 필수)

① LOO+컷오프 측정 (§4) → ② 포지션 기대치 vs 프레임워크 표 대조 (R이면 저점이 정상일 수 있음)
→ ③ 파생 재점검: 원문 각주·인용을 실측 + scan_citation_candidates로 미학습분 체크리스트 등재
→ ④ 검증층 판독: 결과 json의 statute_findings·statute_regen·cite_existence 확인
→ ⑤ 장부 마감(eval 후보 섹션에서 제거·결과 기록) + 커밋.
**배치라도 건별로 ③④를 생략하지 말 것** (생략했다가 실손실 확인된 전례: 20-0008 cov 0.5 = 파생 3건 미학습).

## 4. 측정 코드 스니펫 (복붙용 — Git Bash heredoc)

### 4-1. 해석례 직접 eval (LOO+컷오프)
질문=해석례의 질의요지, 정답=회답·이유 요지, 컷오프=회신일 전날.
```bash
cd "/d/00 USUN AX/## Workspace(model4, April, Gemini)" && PYTHONIOENCODING=utf-8 python - <<'EOF'
import sys, json, importlib.util
sys.stdout.reconfigure(encoding="utf-8")
spec = importlib.util.spec_from_file_location("ev", "scripts/eval_pipeline.py")
ev = importlib.util.module_from_spec(spec); spec.loader.exec_module(ev)
q  = "<질의요지를 질문문으로>"
gt = "<회답+이유 요지>"
gen = ev.get_generator()
r = gen.generate(query=q, provider="gemini",
                 as_of_date="YYYY-MM-DD",        # 회신일 전날
                 as_of_code="XX-XXXX",           # 안건번호 (시제·역할 leakage 필터에 쓰임)
                 exclude_doc_codes={"XX-XXXX"})  # 자기 자신 제외 (LOO)
ans = r.get("answer","")
j = ev.run_judge(q, gt, ans, [d.metadata.get("doc_code","") for d in r.get("qa_docs",[])], provider="gemini")
print(f"corr={j.get('correctness')} cov={j.get('coverage')} gap={j.get('gap_type')} spurious={j.get('spurious_cite')}")
print("missing:", j.get("missing_articles"), j.get("missing_precedents"))
json.dump({"code":"XX-XXXX","mode":"loo","as_of_date":"YYYY-MM-DD","question":q,"ground_truth":gt,
           "system_answer":ans,"scores":j},
          open("data/eval_results/XX-XXXX_eval.json","w",encoding="utf-8"), ensure_ascii=False, indent=1)
EOF
```
판례는 동일 패턴으로 **간접 측정**: 질문=판시사항을 질문문으로, 정답=판결요지, 컷오프=선고 전날,
결과 파일명 `<사건번호>_indirect_eval.json`.

### 4-2. 판독 기준
- `gap=none`은 "실패 분류 없음"(corr≥0.7)이지 인용 완전 일치가 아니다.
- LOO에서 `spurious_cite=True`인데 답이 "존재여부 리서치 필요"로 유보했다면 → 컷오프가 리딩 판례를
  차단한 **LOO 인공물**일 수 있음. **비-LOO(인자 3개 빼고 재생성)로 대조**해서 판별하라 — 비-LOO에서도
  같은 흠이 나오면 코퍼스 공백(진짜 보수 대상), 사라지면 인공물(보수 불요). 2015두41579 전례.
- ⑥b < ⑥a면 라이트트랙 재료 자체를 의심 (§6).

```
├─ 계열은 있는데 어휘 직교로 안 불림 → 층위 판별이 관건:

```

## 6. 라이트트랙 함정 2건 (22-0293 실증 — ⑥b가 ⑥a보다 나빠진 실제 사례)

1. **L(입법 실효) 재료**: 구법 '개별 결론'을 그 도메인 어휘로 배선하면 현행 질의를 오염시킨다.
   L에 살아있는 일반 논지가 있으면 **그 논지 어휘로** 재큐레이션해 배선(06-0115: '공개공지 완화'→
   '위임 구조'로 재배선해 회복). 순수 L은 입법연혁 서술로만.
2. **판정자 지목 선례(missing_precedents)가 방해일 수 있다**: 반대 방향 뉘앙스면 원류 논지를
   흐린다. ⑥b 하락 시 재료를 빼고 재측정해볼 것.

## 7. 판정 3축 + 레코드 규격

- **축1 포지션**: R(원류)/M(방법론)/A(적용)/B(경계)/X(긴장)/L(입법승격). 복합 표기 가능(A+B, R+L).
  확정은 사용자 공동(§0-1).
- **축2 배선**: 레코드 검색 / 문형 훅 / 원칙 P-0XX / 조문 프레임 article_roles / 판례 doctrine.
- **축3 티어**: T1 판례 / T2 법제처 / T3 중앙부처 회신 / T4 지자체 회신.
  T3/T4의 가치는 권위가 아니라 **"실제로 이렇게 결정된 유사 사례가 있다"는 확인 기능**.
  규칙: ① 레코드에 `tier` 필드, ② cite_label에 출처집 표기(예: `…회신/서울시질의회신집`),
  ③ **반드시 근거 조문 프레임에 role_sources(type:실무회신)로 연결**, ④ PASS2가 "실무상 유사
  사례로, ~" 형태로 분리 제시(모순 시 상위 티어 우선).
- **판례 doctrine 블록** (court_cases 레코드 필수):
  `{"position": "R", "series": "<계열명 — 기존 계열과 정확히 같은 문자열이어야 연결됨>",
    "scope": "횡단|도메인", "doctrine_terms": ["<법리어휘 3~5개, 범용 토큰 금지>"], "principle_ref": null}`
- **논지어휘(doctrine_terms) 판정 기준 (2026-07-21 연구 확립)**: 소환은 문자 포함 매칭이므로
  어휘 선택 = "어떤 질문 문장이 이 판례를 부를 수 있는가"의 정의다.
  - **정의**: 논지어휘 = 그 계열의 논지 질문자가 **반드시 쓰게 되는** 어휘. 사안어휘 = 이 사건의
    사실관계에 있을 때만 쓰는 어휘 — 금지.
  - **차기 사건 테스트**: "이 계열의 다음 판례 사안에도 이 어휘가 등장하는가?" 등장하면 논지어휘,
    아니면 사안어휘. (실사고 소급 검증 3건 전부 차단: 2017두73693 '심의생략' 태그 / 06-0115
    '공개공지 완화' 배선(22-0293 오염) / 91누8319 오발화.)
  - **제도 명사 ≠ 사안어휘**: 제도 자체가 계열(여러 법 횡단)이면 계열구성어휘로 정당
    (신탁재산·이행강제금·재량행위 ✓). 판별축은 명사 종류가 아니라 **계열을 정의하느냐,
    이 사건을 정의하느냐**.
  - **2층 분리**: 판례는 자기 논지어휘로만 소환되고, 순수 도메인 질의는 원칙 P-0XX가 논지를
    실어 나른다 — 도메인 어휘를 판례에 태깅해 억지 소환시키지 말 것(케이스패치).
  - 양방향 가드: 너무 넓으면(_DOCTRINE_STOP·lint 하한) / 너무 좁으면(위 테스트, 수동 상한).
    코퍼스 전이성 스캔은 참고용 — 얇은 계열에서 오탐(재량·증명책임 실측)하므로 차단 근거 금지.
- 조문 프레임 스키마는 `data/article_roles/건축법_제38조.json`(계열해 배선 표준례)·
  `건축법_제11조.json`(대척 논지 병치 표준례)을 모방.

## 9. 알려진 함정 (재발 방지 목록)

- REINDEX는 PYTHONIOENCODING 없으면 조용히 실패 (§2⑧).
- Chroma where 필터에 `$contains` 없음 — 쓰면 무필터 폴백. Python 후처리로 거를 것.
- 증분 add된 doc은 HNSW search_ef 문제로 검색 누락 가능 — 전량 재인덱싱이 정석.
- `search_cases_by_doctrine`는 `Retriever`가 아니라 `Retriever()._searcher`(HybridSearcher)의 메서드.
- make_ingest_report·check_temporal_drift는 `법제처_*` 파일명만 지원 — 판례·T3/T4는 수동(백로그).
- doc_date "XX" 포함·공백이면 시제 필터가 오작동 — curate self-test가 잡는다.
- 원칙(P-0XX) 승격은 일반해만 — 계열해를 승격하려다 철회한 전례(P-014). 헷갈리면 사용자에게.
- 판독 중 relation_types가 빈 리스트일 때 doctrine 경로가 죽었던 게이트 버그는 수정됨(21-0186) —
  유사한 "경로가 게이트 안에 있어 조기 return으로 죽는" 패턴을 보수 시 경계할 것.
