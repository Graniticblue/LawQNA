# 건축법규 검토 RAG 시스템 — 판례 파이프라인 종합 계획

## 1. 프로젝트 개요

### 1.1 목표
기존 **국가법령 + 질의회신 RAG** 시스템에 **판례(판결문) RAG**를 추가하여,
법규 검토 시 관련 판례까지 자동 검색·인용할 수 있는 3층 검색 체계를 구축한다.

### 1.2 최종 아키텍처 (2단계 라우팅 + 3층 검색)

```
[사용자 질문]
    │
    ▼
╔══════════════════════════════════════════════════════╗
║  1차 LLM 호출 — 쟁점 식별 + 2단계 라우팅 트리거           ║
║                                                      ║
║  출력 ①: 관련 법령 힌트 (법명, 조문번호)                 ║
║  출력 ②: 판례-법령 관계 유형 분류 (7가지 중 1~N개 중첩)   ║
║          DEF_EXP | SCOPE_CL | REQ_INT | EXCEPT       ║
║          INTER_ART | PROC_DISC | SANC_SC             ║
╚══════════════════════════════════════════════════════╝
    │                           │
    ▼                           ▼
[라우팅 1단계: 법령 특정]     [라우팅 2단계: 유형 분류 (복수 가능)]
 ├─ Layer 1: 토픽 기반        "이 질문은 7가지 유형 중
 ├─ Layer 2: keyword_law_map   어디에 해당하는가? (1~N개)"
 └─ Layer 3: 조문 그래프       → 예: [DEF_EXP, REQ_INT]
    │                           │
    └──────────┬────────────────┘
               ▼
    ┌─ (법규 × 유형) 쌍 도출 ──────────────┐
    │  복수 유형 시 조합 쌍이 여러 개 생성    │
    │  예: 건축법 제2조 × DEF_EXP           │
    │      건축법 제2조 × REQ_INT           │
    └────────────┬─────────────────────────┘
                 │
                 ▼
[하이브리드 검색 (Vector + BM25 → RRF)]
 ├─ 1층: law_articles     법규 필터로 조문 검색       ← 기존
 ├─ 2층: qa_precedents    유사 질의회신 검색          ← 기존
 └─ 3층: court_cases      (법규 × 유형) 쌍 필터로     ← 신규
                           해당 유형(들)의 판례 OR 검색
                 │
                 ▼
[2차 LLM 호출] → 조문 + 질의회신 + 유형 매칭 판례 기반 법리 답변
  컨텍스트 = (법규 조문) + (관계 유형) + (유사 판례)
```

**핵심 원리**: 판례는 단순히 "의미적으로 비슷한" 것을 찾는 게 아니라,
**같은 법규를 같은 방식(유형)으로 해석한 판례**를 찾아야 한다.
→ (법규 × 유형) 쌍이 판례 검색의 핵심 키가 된다.
→ **하나의 질문이 복수 유형에 해당할 수 있으므로**, 복수 (법규 × 유형) 쌍이
  동시에 도출되며, 각 쌍별로 독립 검색 후 결과를 병합한다.

## 2. Generator 통합 — 2차 LLM 컨텍스트 포맷

### 2.1 현재 → 개선 비교

| 항목 | 현재 | 개선 |
|------|------|------|
| 조문 | `[1] 건축법 제2조 ...` | 동일 (변경 없음) |
| 질의회신 | `content[:500]` (잘림) | `chunk_answer` 전체 + 메타 |
| 판례 | 없음 | `[참조 판례 풀]` (신규) |

### 2.2 개선 컨텍스트 포맷 (판례 포함)

```
=== [관련 법령 조문] ===
[1] 건축법 제2조 (정의)
{조문 내용}

=== [유사 질의회신 선례] ===
[선례1] {question_type} | 확신도: {verdict}
질문: {question}
{chunk_answer 전체}

=== [참조 판례 풀] ===
아래는 (법규 × 유형) 쌍으로 검색된 관련 판례입니다.
법리 해석 중 조문만으로 판단이 애매한 지점에서만 인용하세요.
확장된 정의를 원용할 경우, 해당 판례의 조건이 본 건과
일치하는지 반드시 검토하세요.

[판례1] 대법원 2023두12345 (2023.5.15. 선고)
  유형: 🔵 정의확장형 | 법규: 건축법 제2조제1항제1호
  사실 요약: 분할예정 토지의 대지 해당 여부
  논리 요약: 문언보다 입법 취지 중심으로 넓게 해석
  요지: "대지"의 범위에 분할예정 토지 포함
  적용 조건: 사용승인 시 분할이 예정된 경우에 한함
```

### 2.3 Pass 2 답변 구조

```
[쟁점 식별] → [관련 조문 확인] → [사실관계 분기]
→ [법리적 판단 로직]    ← 해석 분기점에서 판례 인용 (Section 3 참조)
→ [최종 답변] → [근거 법령 + 인용 판례] → [검색 태그]
```

## 3. 판례 활용 전략 — 3가지 모드

### 3.1 기본 원칙 — 하이브리드 방식

판례는 (법규 × 유형) 쌍으로 **미리 확보**하되 (선제 투입),
LLM이 법리 해석 중 **판단이 애매한 지점에서만 정밀 인용**한다 (사후 검증).

### 3.2 3가지 활용 모드

| 모드 | 이름 | 핵심 | 매칭 조건 | 답변 내 위치 |
|------|------|------|----------|-------------|
| ① | **직접 적용** | 같은 상황의 판례 | 법령 ✅ 유형 ✅ 키워드 ✅ | `[관련 판례 검토]` 별도 섹션 |
| ② | **논리 차용** | 판단 방법론을 빌려옴 | 법령 ✅ 유형 ✅ 키워드 ❌ | `[법리적 판단]` 내 인라인 |
| ③ | **정의 원용** | 확장된 정의를 전제로 사용 | 선행 판례의 정의가 전제 조건 | `[관련 조문 확인]` 직후 |

**모드 ① 직접 적용** — "같은 예"
- 판례가 지금 질문과 거의 같은 상황을 다뤘을 때
- 가장 강한 인용: "이 판례가 선례입니다"
- `fact_summary` 벡터 유사도가 높을 때 해당

**모드 ② 논리 차용** — "논리를 빌려쓰는 예"
- 다른 쟁점이지만 같은 해석 방법론(판단 틀)을 적용할 수 있을 때
- "이 판례의 논리에 비추어 ~"
- `logic_summary` 벡터 유사도가 높을 때 해당

**모드 ③ 정의 원용** — "확장된 정의를 가져다 쓰는 예"
- 선행 판례가 넓혀놓은 정의가 지금 논증의 **전제 조건**일 때
- "판례에 의해 ~로 확장되었으므로, 이를 전제로 ~"
- ⚠️ **조건부 적용 검증 필수**: 판례가 확장한 정의에는 적용 조건이
  붙어있을 수 있음 (예: "사용승인 시에 한하여"). 본 건의 사실관계가
  해당 조건을 충족하는지 반드시 확인해야 함.

### 3.3 답변 예시 (모드 표기 포함)

```
### [쟁점 식별]
쟁점: 분할예정 토지에 건축허가가 가능한지

### [관련 조문 확인]
① 건축법 제2조제1항제1호: "대지"란 ... 토지를 말한다
② 건축법 제11조: 건축허가를 받으려면 대지에 ...

### [정의 원용]                                    ← 모드 ③
📌 대법원 2023두12345: "분할예정 토지도 대지에 해당"
   적용 조건: 사용승인 시 분할이 예정된 경우
   → 본 건 사실관계와 일치 → 전제로 채택

### [법리적 판단 로직]
1. 대지 해당 여부: ✅ (모드 ③에 의해 확정)
2. 건축허가 요건: 제11조 각호 검토
3. ⚠️ 해석 분기점: [특정 요건] 충족 여부 불명확
   📌 유사 판례의 논리 차용 (모드 ②)               ← 모드 ②
   → 해석 확정

### [최종 답변]
[확신도: 확정]
가능합니다. ...

### [근거 법령 + 인용 판례]
- 「건축법」 제2조제1항제1호, 제11조
- 대법원 2023. 5. 15. 선고 2023두12345 판결
```

### 3.4 유형별 판례 인용 지침

| 유형 | 해석 분기점 | 판례 활용 방식 |
|------|-----------|---------------|
| `DEF_EXP` | "~까지 포함하는가?" 불명확 | 확장된 정의 범위 제시 |
| `SCOPE_CL` | "~에 적용되는가?" 불명확 | 확정된 적용 경계 제시 |
| `REQ_INT` | "~를 충족하는가?" 불명확 | 구체화된 충족 기준 제시 |
| `EXCEPT` | "예외가 인정되는가?" 불명확 | 인정된 예외 조건 제시 |
| `INTER_ART` | "어느 조문이 우선?" 불명확 | 확립된 우선순위 제시 |
| `PROC_DISC` | "재량 범위는?" 불명확 | 확인/제한된 범위 제시 |
| `SANC_SC` | "제재 범위는?" 불명확 | 구체화된 기준 제시 |

### 3.5 확신도 판정 + 후속 질문 생성

시스템은 **법리적으로 최선의 답변**을 내놓되,
확정할 수 없는 부분은 **담당부서에 확인할 구체적 질문**을 함께 제시한다.

**확신도 4단계**:

| 확신도 | 의미 | 시스템 행동 |
|--------|------|-----------|
| **확정** | 법+판례로 결론 명확 | 확정 답변 제공 |
| **조건부** | 사실관계에 따라 결론 상이 | 조건별 답변 + 확인 질문 |
| **재량위임** | 허가권자 재량 사항 | 법적 근거 제시 + 확인 질문 |
| **범위외** | 시스템 검토 범위 밖 | 관할부서 안내 + 확인 질문 |

**"문의바랍니다" 트리거 5가지**:

1. **재량 판단 영역** — 법/판례로 답이 정해지지 않는 허가권자 재량
2. **현장별 사실관계 필요** — 시스템이 알 수 없는 현장 정보
3. **관련 조문·판례 부재** — 검색 결과 없음
4. **판례 충돌** — 상반된 판례 존재
5. **복합 조건 미해결** — 분기 조건이 너무 많아 확정 불가

**답변 구조 (확정이 아닌 경우)**:

```
### [최종 답변]
[확신도: 조건부]
법리적 검토 결과, 가장 유력한 답변은 다음과 같습니다:
{최선의 분석 결과}

다만, 아래 사항은 관할 건축허가 담당부서에 확인이 필요합니다:

### [담당부서 확인 질문]
아래 질문을 관할 구청 건축과에 문의하시기 바랍니다:
1. "해당 필지의 용도지역이 [X]에 해당하는지?"
   → 이에 따라 건폐율 기준이 달라집니다
2. "기존 건축물의 용도변경 이력이 있는지?"
   → 이력에 따라 제11조 적용 여부가 달라집니다
3. "분할 조건부 허가에 대한 귀 구청의 행정 관행은?"
   → 재량 영역으로 관할청마다 처리 기준이 다를 수 있습니다
```

**핵심**: 단순히 "문의바랍니다"가 아니라, **어떤 질문을 왜 해야 하는지**까지 제시.
각 질문 아래에 `→`로 그 질문이 답변에 미치는 영향을 설명한다.

## 4. 검색 통합 (법규 × 유형 쌍 기반)

### 4.1 쿼리 유형 분류 (1차 LLM 호출에 포함)

1차 LLM 호출 시 쟁점 식별과 함께 **질문의 관계 유형**도 분류한다.
이 유형은 판례 검색의 핵심 필터로 사용된다.

**1차 LLM 출력 스키마 (확장)** — `relation_types`는 배열로 복수 유형 중첩 지원:
```json
{
  "issue": "건축법상 '대지'에 분할 예정 토지 포함 여부 및 이에 따른 건축허가 요건",
  "question_type": "복수조문탐색형",
  "law_hints": ["건축법 제2조", "건축법 제11조"],
  "relation_types": [
    {"type": "DEF_EXP", "reason": "'대지' 용어 정의의 범위를 묻고 있음", "weight": 1.0},
    {"type": "REQ_INT", "reason": "건축허가 요건 충족 여부도 함께 묻고 있음", "weight": 0.7}
  ]
}
```

> **복수 유형 중첩 설계 원칙**:
> - 하나의 질문이 여러 법적 쟁점을 동시에 포함할 수 있으므로, 유형 분류는 **단일 해가 아닌 1~N개의 중첩 해**를 허용한다.
> - 각 유형에는 `weight` (0.0~1.0)를 부여하여, 주 쟁점(primary)과 부 쟁점(secondary)을 구분한다.
> - `weight ≥ 0.5`인 유형만 판례 필터링에 사용하고, `weight < 0.5`인 유형은 RRF 랭킹 시 부스트 계수로만 활용한다.
> - 예시:
>   - *"'대지'에 분할예정 토지가 포함되는지?"* → `[DEF_EXP(1.0)]` (단일)
>   - *"분할예정 토지에서 건축허가가 가능한지?"* → `[DEF_EXP(1.0), REQ_INT(0.7)]` (중첩)
>   - *"이행강제금 부과 시 용도변경 예외가 인정되는지?"* → `[SANC_SC(1.0), EXCEPT(0.8)]` (중첩)

**유형 분류 기준 (1차 LLM 프롬프트에 포함)**:

| 유형 | 질문의 성격 | 질문 예시 |
|------|-----------|----------|
| `DEF_EXP` | 법 용어의 정의·범위를 묻는 질문 | "~에 ~도 포함되나요?" |
| `SCOPE_CL` | 조문이 특정 상황에 적용되는지 묻는 질문 | "~에도 ~이 적용되나요?" |
| `REQ_INT` | 요건 충족 여부·기준을 묻는 질문 | "~의 요건은?", "~에 해당하나요?" |
| `EXCEPT` | 예외·특례 적용 가능성을 묻는 질문 | "~임에도 불구하고 ~가 가능한가요?" |
| `INTER_ART` | 조문 간 충돌·우선순위를 묻는 질문 | "~와 ~가 충돌하면 어떤 것을?" |
| `PROC_DISC` | 절차·재량 범위를 묻는 질문 | "허가권자의 재량 범위는?" |
| `SANC_SC` | 위반 제재·벌칙 범위를 묻는 질문 | "과태료가 얼마?" |

### 4.2 판례 검색 전략 — (법규 × 유형) 쌍 필터링

**기존 `05_Retriever.py` 확장 (모듈 분리 검토 필요)**

```python
def retrieve(
    self, query, question_type,
    relation_types,             # ← 1차 LLM이 분류한 유형들 (복수, 신규)
                                #    예: [{"type": "DEF_EXP", "weight": 1.0},
                                #         {"type": "REQ_INT", "weight": 0.7}]
    top_k_law=7, top_k_qa=3, top_k_case=5
):
    """
    relation_types가 복수일 경우:
    - weight ≥ 0.5인 유형들로 각각 필터 검색 실행
    - 결과를 RRF로 병합하되, weight를 부스트 계수로 활용
    - top_k_case를 늘려 다양한 유형의 판례가 포함되도록 함

    Returns:
        law_docs: List[RetrievedDoc]     ← 기존 (법규 필터 검색)
        qa_docs: List[RetrievedDoc]      ← 기존
        case_docs: List[RetrievedDoc]    ← 신규 (법규×유형 복수 필터 검색)
    """
```

**판례 검색 흐름** (복수 유형 대응):

```
1차 LLM 출력:
  law_hints = ["건축법 제2조", "건축법 제11조"]
  relation_types = [
    {"type": "DEF_EXP", "weight": 1.0},   ← 주 유형
    {"type": "REQ_INT", "weight": 0.7}     ← 부 유형
  ]
         │
         ▼
[court_cases 컬렉션 — 유형별 병렬 검색]
         │
         ├─ 검색 ①: DEF_EXP (weight=1.0)
         │   ├─ 필터: cited_laws_str ∋ "건축법" AND relation_types ∋ "DEF_EXP"
         │   ├─ Vector + BM25 → RRF
         │   └─ 결과에 weight=1.0 부스트 적용
         │
         ├─ 검색 ②: REQ_INT (weight=0.7)
         │   ├─ 필터: cited_laws_str ∋ "건축법" AND relation_types ∋ "REQ_INT"
         │   ├─ Vector + BM25 → RRF
         │   └─ 결과에 weight=0.7 부스트 적용
         │
         ├─ 결과 병합: 두 검색 결과를 RRF 재병합
         │   ├─ 중복 판례는 최고 가중 점수 채택
         │   └─ 상위 top_k_case개 선택
         │
         └─ Fallback 전략:
             ├─ 특정 유형 결과 < 2건 → 해당 유형 필터 완화 (법규만)
             ├─ 전체 결과 < top_k_case → 유형 필터 전부 완화
             └─ 여전히 부족 → 법규 필터도 완화, 전체 검색
```

**Fallback 전략이 필요한 이유**:
- 초기 판례 데이터가 적을 때 (법규 × 유형) 쌍이 비어있을 수 있음
- 예: 건축법 제2조 + EXCEPT 조합의 판례가 없을 수 있음
- 이 경우 유형 필터를 풀고 법규 기준으로만 검색 → 없으면 전체 검색
